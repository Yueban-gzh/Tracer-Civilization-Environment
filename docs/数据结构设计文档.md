# 数据结构设计文档

## 概述

本项目是一款卡牌策略RPG游戏（类Slay the Spire），采用C++17 + SFML 3.0.2开发。文档以**数据结构**为核心，全面梳理项目中使用的数据模型、存储结构及其相互关系。

---

## 一、核心数据类型

### 1.1 标识符类型（Common/Types.hpp）

所有实体统一使用 `std::string` 作为标识符，便于JSON序列化与哈希表索引：

```cpp
using CardId = std::string;        // 卡牌ID（如 "strike", "strike+"）
using MonsterId = std::string;     // 怪物ID（如 "1.1_xiuqicanzun"）
using StatusId = std::string;      // 状态ID（buff/debuff）
using PotionId = std::string;      // 药水ID
using RelicId = std::string;       // 遗物ID
using InstanceId = int;            // 实例ID（牌组内唯一，区分同名卡牌）
```

### 1.2 枚举类型

#### 卡牌类型（CardType）
```cpp
enum class CardType { 
    Attack,    // 武学牌：造成伤害
```

## 二、静态数据层（DataLayer）

### 2.1 卡牌静态数据（CardData）

**存储位置**: `data/cards.json`  
**数据结构**: `tce::CardData`

```cpp
struct CardData {
    CardId     id;                    // 卡牌唯一标识
    std::string name;                 // 显示名称
    CardType   cardType;              // 卡牌类型
    int        cost;                  // 费用（-1表示X费）
    CardColor  color;                 // 颜色
    Rarity     rarity;                // 稀有度
    std::string art;                  // 立绘路径
    std::string description;          // 描述文本
    bool       exhaust;               // 消耗：打出后移入消耗堆
    bool       ethereal;              // 虚无：回合末未打出则消耗
    bool       innate;                // 固有：开局必在手牌
    bool       retain;                // 保留：回合末不弃牌
    bool       unplayable;            // 不可打出
    bool       requiresTarget;        // 是否需要选择目标
    bool       irremovableFromDeck;   // 不可从牌组移除
};
```

**JSON示例**：
```json
{
```

## 三、战斗运行时数据结构

### 3.1 卡牌实例（CardInstance）

**文件**: `CardSystem/CardSystem.hpp`

```cpp
struct CardInstance {
    InstanceId instanceId;         // 运行时唯一标识
    CardId     id;                 // 对应静态CardData的ID
    bool       temporary;          // 是否临时生成
    bool       combat_cost_zero;   // 本场战斗耗能视为0
    int        combatCostDiscount; // 累计减费

```

**设计要点**:
- `instanceId` 区分同名卡牌（如2张"穿云箭"）
- `temporary` 标记用于牌组视图过滤
- 战斗结束后临时牌自动销毁

### 3.2 四牌堆系统

**文件**: `CardSystem/CardSystem.hpp`

```cpp
class CardSystem {
private:
    std::vector<CardInstance> master_deck_;    // 永久牌组（跨战斗）
    std::vector<CardInstance> draw_pile_;      // 抽牌堆
    std::vector<CardInstance> hand_;           // 手牌
    std::vector<CardInstance> discard_pile_;   // 弃牌堆
    std::vector<CardInstance> exhaust_pile_;   // 消耗堆
    
    std::unordered_map<CardId, std::function<void(EffectContext&)>> effect_registry_;
};
```

**数据结构关系**:
```
master_deck_ (永久牌组)
    ↓ 战斗开始时
draw_pile_ → hand_ → discard_pile_ → (洗牌) → draw_pile_
                  ↓
              exhaust_pile_ (不可回收)
```

**关键约束**:
- 手牌上限: `hand_limit_ = 10`
- 抽牌为空时自动洗弃牌堆
- 消耗堆不参与洗牌

### 5.1 事件决策树

**文件**: `EventEngine/EventEngine.hpp`

事件系统采用**树形结构**，支持选项跳转：

```
event_002 (鎏金神像)
├─ 选项A: 捧起神像 → event_002a
│   ├─ 选项A1: 以身承压 → [max_hp-6, relic+1]
│   ├─ 选项A2: 以血换气 → [heal-12, relic+1]
│   └─ 选项A3: 弃像于地 → [add_curse+1, relic+1]
└─ 选项B: 不触因果 → [none]
```

### 5.2 遍历算法

```cpp
void traverse_preorder(EventId root_id, std::function<void(const Event&)> visit) const;
void traverse_level_order(EventId root_id, std::function<void(const Event&)> visit) const;
```

- **先序遍历**: 深度优先访问事件树

## 九、数据流与模块关系

### 9.1 模块依赖图

```
┌─────────────┐
│  DataLayer  │ (静态数据)
│  cards.json │
│  monsters.json│
│  events.json │
└──────┬──────┘
       │ 只读
       ├──────────────┬──────────────┐
       ↓              ↓              ↓
┌──────────┐  ┌──────────┐  ┌──────────┐
│CardSystem│  │BattleCore│  │EventEngine│
│  (C模块) │  │  (B模块) │  │  (D模块) │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │              │
     └─────────────┼──────────────┘
                   ↓
          ┌────────────────┐
          │GameFlowController│
          │  (主流程控制)    │
          └────────┬───────┘
                   ↓
          ┌────────────────┐
          │  SaveSystem    │
          │  (存档/读档)    │
          └────────────────┘
```

### 9.2 数据生命周期

```
游戏启动
  ↓
加载静态数据 (JSON → CardData/MonsterData/Event)
  ↓
角色选择 → 初始化 master_deck
  ↓
生成地图 (MapConfig → MapNode图)
  ↓
┌─ 地图界面 → 选择节点
│     ↓
│   进入节点 (保存检查点)
│     ↓
│   ┌─ 战斗节点 → BattleState → 战斗结算 → 奖励
│   ├─ 事件节点 → Event树 → 选项 → 效果结算
│   ├─ 商店节点 → ShopDisplayData → 购买/删除
│   ├─ 休息节点 → RestDisplayData → 回血/升级
│   └─ 宝藏节点 → TreasureRoomOutcome → 奖励
│     ↓
│   返回地图 (标记节点完成)
└─ 循环直到Boss战胜利
  ↓
游戏结束/存档
```

## 十、数据结构设计模式

### 10.1 策略模式

**应用**: 地图配置（MapConfig）
- 基类定义接口
- 派生类实现不同地图布局
- 运行时通过 `MapConfigManager` 切换

### 10.2 快照模式

**应用**: 战斗状态、存档系统
- `BattleState` → `BattleStateSnapshot` (UI展示)
- 检查点机制防止SL作弊
- 序列化/反序列化JSON

### 10.3 注册表模式

**应用**: 卡牌效果系统
```cpp
std::unordered_map<CardId, std::function<void(EffectContext&)>> effect_registry_;
```
- 卡牌ID映射到效果函数
- 动态注册，便于扩展
- 查询时间复杂度 O(1)

### 10.4 回调函数注入

**应用**: 模块解耦
```cpp
using GetCardByIdFn = std::function<const CardData*(CardId)>;
using AddToMasterDeckFn = std::function<void(CardId)>;
```
- 避免循环依赖
- 便于Mock测试
- 提高模块独立性

| 场景 | 数据结构 | 原因 |
|------|---------|------|
| 卡牌效果查找 | `unordered_map` | O(1)查询 |
| 手牌/牌堆 | `vector` | 缓存友好，顺序访问 |
| 静态数据索引 | 全局数组 + ID映射 | 快速访问 |
| 地图节点图 | 邻接表（prev/next数组） | 灵活遍历 |

### 11.2 内存管理

- **静态数据**: 游戏启动时一次性加载，常驻内存
- **卡牌实例**: 使用 `InstanceId` 区分，战斗结束回收
- **临时牌**: `temporary=true` 标记，自动清理
- **快照**: 仅在存档/检查点时创建

### 11.3 序列化优化

- 坐标压缩: `float * 100 → int` 减少精度损失
- 枚举转字符串: 提高JSON可读性
- 版本号: `schema_version` 支持向后兼容
