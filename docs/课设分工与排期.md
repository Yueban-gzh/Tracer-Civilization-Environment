# 《溯源者：文明环境》课设 —— 分工与排期

> 分工、排期、每人每阶段事项与阶段成果。设计细节与接口见 [设计与接口.md](设计与接口.md)。

---

## 一、项目与课设要求（摘要）

- **游戏**：《溯源者：文明环境》国风爬塔卡牌，对标杀戮尖塔。MVP：一局跑通「地图→选节点（战斗/事件/休息/商店）→战斗用牌→奖励→下一层→Boss 结算」。
- **课设数据结构**：线性表/链表（牌堆）、栈（回合结算）、队列（行动顺序）、树（事件树）、图（地图）、查找（哈希/BST）、排序（奖励/排行榜）。报告需覆盖 80%+。
- **架构**：Data 层（表+查找）→ Core（Map/Battle/Event）→ UI。

---

## 二、五人分工


| 角色    | 模块       | 数据结构落点                        |
| ----- | -------- | ----------------------------- |
| **A** | 地图       | 图、BFS/DFS、路径与可达性              |
| **B** | 战斗引擎     | 栈（回合结算）、队列（行动顺序/日志）           |
| **C** | 卡牌系统     | 线性表/链表（deck/hand/discard，抽弃洗） |
| **D** | 事件/商店/休息 | 树（事件决策树、遍历）                   |
| **E** | 数据层      | 哈希/BST 查找、排序、表数据              |


**依赖**：E 被 A/C/D 依赖；C、E 被 B 依赖；D 与 C、E 交互。主流程调 A/B/D。

## 三、各阶段事项与成果

### 阶段 0（第 1 周）：需求与接口约定

**本阶段成果**：《设计与接口》中 MVP 范围、数据表格式、公共类型、各模块对外接口及依赖全部定稿；全员按同一份约定开发，不再私下改接口。


| 谁   | 本阶段要做的事                                                                                                                                                       |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 全员  | 一起定 MVP：地图层数、每层节点数、牌数/敌人数、是否做商店/事件树；画清一局流程。                                                                                                                   |
| E   | 起草卡牌表、怪物表、事件表字段（含类型、示例），交全员评审后定稿。                                                                                                                             |
| 全员  | 定公共类型：NodeType、CardId、MonsterId、EventId、NodeId。                                                                                                               |
| A   | 写清本模块对外接口（init_map、get_nodes_at_layer、get_next_layer、is_boss_node、validate_reachable 等）及入参/出参；声明不依赖他人或仅可选依赖 E。                                                |
| B   | 写清本模块对外接口（start_battle、play_card、end_turn、is_battle_over、get_reward_cards 等）及需要 C、E 提供的接口列表。                                                                  |
| C   | 写清本模块对外接口（init_deck、draw_cards、get_hand、remove_from_hand、add_to_discard、shuffle、add_to_deck、remove_from_deck、upgrade_card_in_deck 等）及需要 E 提供的 get_card_by_id。 |
| D   | 写清本模块对外接口（start_event、choose_option、open_shop、buy_card、rest_heal、rest_upgrade 等）及需要 E、C 提供的接口列表。                                                              |
| E   | 写清本模块对外接口（get_card_by_id、get_monster_by_id、get_event_by_id、sort_cards_by_rarity、load_* 等），无依赖。                                                                |


---

### 阶段 1（第 2 周）：数据层与基础结构

**本阶段成果**：E 提供可用的表与按 id 查找；A 能生成地图并做 BFS/DFS 可达性；C 牌堆可抽/弃/洗；D 至少 1 个事件树可遍历；B 用 Mock C、Mock E 能跑通一局战斗流程（可单测）。五人可并行，依赖方未完成时用 Mock。


| 谁   | 本阶段要做的事                                                                                                                                                                                                          |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| E   | 按《设计与接口》填好卡牌/怪物/事件表数据（满足 MVP 数量）；实现 get_card_by_id、get_monster_by_id、get_event_by_id（哈希或 BST）。                                                                                                                   |
| A   | 实现图结构（邻接表或矩阵）、地图生成（固定层数、每层节点数，DAG）、get_nodes_at_layer、validate_reachable（BFS/DFS）；不依赖 B/C/D，节点类型可写死或从常量读。                                                                                                        |
| C   | 实现 deck/hand/discard（线性表或链表）、init_deck、draw_cards、get_hand、remove_from_hand、add_to_discard、shuffle_discard_into_draw、add_to_deck、get_deck_size、get_discard_size；需 E 的 get_card_by_id 时若 E 未就绪则自写 Mock E 返回固定牌数据。 |
| D   | 定义事件树节点结构，实现 1～2 个示例事件（选项→结果），支持先序/层序遍历；需 E 的 get_event_by_id 时若 E 未就绪则自写 Mock E 返回一棵写死的事件树。                                                                                                                     |
| B   | 实现战斗回合框架：行动队列、出牌入栈、回合末出栈、胜负判定；内部调 C（抽牌/出牌/弃牌）、E（查怪物）。C、E 未就绪则自写 Mock C、Mock E（见《设计与接口》第六节），先跑通回合逻辑。                                                                                                              |


---

### 阶段 2（第 3～4 周）：核心玩法串联

**本阶段成果**：主流程能按层选节点，根据节点类型进入战斗/事件/休息/商店；战斗用真 C、真 E 完成一局；事件/商店/休息能改状态、加牌、删牌、升级牌；奖励可排序展示；从开局到 Boss 结算一局完整跑通，无 Mock。


| 谁   | 本阶段要做的事                                                                                                                                |
| --- | -------------------------------------------------------------------------------------------------------------------------------------- |
| A   | 与主流程对接：get_nodes_at_layer 返回的 nodeType、contentId 能被主流程用于分支（进战斗/事件/休息/商店）；可选已访问节点标记。                                                    |
| C   | 完善与战斗的联动：抽牌堆空时自动洗弃牌堆再抽、出牌扣墨力等；保证 B 仅通过调用 C 的接口即可完成整场战斗。                                                                                |
| B   | 用真 C、真 E 替换 Mock；联调 start_battle、play_card、end_turn、is_battle_over；实现 get_reward_cards，与 E 的排序对接。                                      |
| D   | 实现事件执行（选分支→应用结果→改状态/加牌/金钱）；商店（买牌、删一张牌）；休息（回血 or 升级一张牌）；调用 C 的 add_to_deck、remove_from_deck、upgrade_card_in_deck 及 E 的 get_event_by_id。 |
| E   | 实现 sort_cards_by_rarity（战斗奖励排序）；可选 sort_leaderboard；保证 B、D、主流程能正常调查找与排序。                                                               |
| 主流程 | 串联：按当前层调 A 取节点列表 → 玩家选节点 → 按 nodeType 调 B（战斗）或 D（事件/商店/休息）→ 战斗胜利后奖励经 E 排序后选牌；层推进直至最后一层 Boss 胜后通关结算。（负责人可指定一人或轮流。）                      |

### 阶段 3（第 5～6 周）：演示与报告素材

**本阶段成果**：每个课设要求的数据结构都有可演示的用例（可打印/可调接口验证）；每人完成本模块对应的报告小节与复杂度分析；报告初稿齐备。


| 谁   | 本阶段要做的事                                               |
| --- | ----------------------------------------------------- |
| A   | 实现或输出 BFS 最少步数到 Boss、DFS 所有可达路径；写图与遍历的复杂度分析及报告小节。     |
| B   | 回合栈、行动队列（可选撤销）的日志或接口可展示；写栈/队列的复杂度分析及报告小节。             |
| C   | 抽牌前后/洗牌前后牌库状态可打印，增删改查过程可说明；写线性表/链表的复杂度分析及报告小节。        |
| D   | 事件树先序/层序遍历可输出「所有事件路径」或选项树；写树的遍历与复杂度及报告小节。             |
| E   | 查找过程可展示（按 id 查卡牌/怪物）；排序算法与复杂度可说明；写查找与排序的报告小节；可选存档与统计。 |

## 五、每人职责（一句话）

- **A**：图结构、地图生成、BFS/DFS，给主流程「当前层节点列表、节点类型与 contentId」。
- **B**：战斗回合、行动队列、回合栈、胜负判定；调 C（牌堆）、E（怪物数据）。
- **C**：deck/hand/discard 的抽弃洗、增删、升级牌；调 E（卡牌数据）。
- **D**：事件树执行、商店、休息；调 E（事件数据）、C（加牌/删牌/升级）。
- **E**：卡牌/怪物/事件表、按 id 查找、奖励排序；被 A/B/C/D 调用。
